## 什么问题

**08 版本活多，没有规划好，整个过程很被动**。核心是**想搞清楚怎么避免这种情况再发生**。

**Why:** 被动状态下所有安排都成了应急，主动优化几乎不可能；且团队士气会跟着走。

## 复盘要点（待深挖）

在下次做版本规划前，需要跟自己/团队复盘：

1. **08 版本的活为啥"多"** — 是原本就多、临时加的、还是没估准？
2. **规划哪一步失效了** — 没排、排了没执行、还是排的时候数据不够？
3. **被动的信号是啥** — 什么时候第一次意识到"要来不及了"？下次能不能更早识别？
4. **哪些活其实可以砍/延后/外包** — 08 时被逼着做的活，事后看是必须的吗？

## How to apply（下次版本）

- 版本开始前 **一周** 出**主动式 timeline**（不是被动接需求）
- 每周 review：**已完成 / 在做 / 阻塞** 三栏
- 设**警戒线**：某类任务超过 X 就报警，触发砍/推/换人
- 复盘会不要跳过（哪怕觉得没时间）
