## 背景

2026-07-07 深夜，飞书 AI（天蓝的运营层 AI）主动汇报：**它工作量过载了**。

**问题症状**：
- 手上 13 个常驻 trigger + 1 心跳 = 14 个负担
- trigger #2（菜刀原画同步 V9）+ #13（提交截图收录 X3 群）每天各跑 30 次，菜刀 2026-07-01 沉默后 = 600+ 次 0 命中空转
- 7-7 0:17 翻车就是这俩害的
- 天蓝反馈"觉得它卡卡的" = 太多单点 + 互相打断 → 整体崩

**飞书 AI 自己的诊断**（打通 GitHub 后读到 feedback-fragmented-work.md 之后）：
> 我卡住不是因为我笨，是因为 trigger 太多 + 群消息太碎 + 临时代码翻车多，工作流之间互相打断。

## 决策：A + B 组合，D 不做，C 暂缓

### ✅ A（砍 trigger 数量，成本最低效果最大）

| # | 操作 |
|---|---|
| A1 | #2 菜刀原画同步 V9：**每小时** → **每天 3 次 (9:00 / 14:00 / 18:00)** |
| A2 | #13 X3 群提交截图收录：**完全停**，改被动（菜刀 @ 天蓝时再收） |
| A3 | #6 工作完成回顾 + #1 每日完成推送：**合并**为 22:30 一次出两份 |

trigger 数量从 13 → **12**（不是 13 → 9，飞书自己算术小误后确认）

### ✅ B（砍空转时段，中等成本）

| # | 操作 |
|---|---|
| B1 | HEARTBEAT 心跳巡检：**30 分 → 60 分** |
| B2 | #11 出勤数据维护：**23:17 → 第二天 8:00** 跟 #5 合并（避夜间 base API 限流） |
| B3 | trigger #2 / #13 **保留白天 9-21 全时段**（别过度收缩） |

### ❌ D（不做）—— 守护永久铁律 #5

飞书 AI 曾提议：把"每条说成功都有 tool_call 证据"从全场景 → 只 V9 写操作，省 30-40% 自查成本。

**天蓝拍板拒绝**。理由：
- 铁律 #5 是 2026-06-27 花大代价立起来的（当天翻车 4 次：13:50 P214CA 死婴 / 14:35 4 条假补齐 / 14:01 "已写日记" / 14:24 "MEMORY 改词"）
- **省 30% 自查成本 ≠ 值得重开"假成功"翻车风险**
- 工作量再大也不该拿铁律换

### ⏸ C（暂缓）—— 被动模式先不逼

飞书 AI 提议 C：菜刀沉默期 = 不再扫 X3 群，改"天蓝 @ 我再处理"。

**天蓝拍板暂缓**。理由：
- 需要天蓝改协作习惯（要记得 @）
- 先看 A + B 是否已经让飞书恢复正常，如果 OK 就不用继续压

## 7-14 复盘指标

3 个客观指标看 A + B 效果：

1. **V9 延迟事故**：菜刀说"可以了"到实际录进 V9 的时间。目标：不能因为改为 3 次/天导致延迟超过 3 小时
2. **接住率**：菜刀 @ 天蓝时，30 分钟内接住的比例。目标 ≥ 90%
3. **trigger 空转下降**：#2 从每天 30 次 → 每天 3 次（月对比：600 → 21，**降 96%**）

**7-14 决策树**：
- 3 指标全 OK → B 剩下的项目再上；C 暂时不开
- V9 延迟 > 3 小时 → 把 #2 从 3 次 → 4-5 次
- 接住率 < 90% → 讨论 C（要天蓝改 @ 习惯 vs 加更多主动巡检）

## 关键洞察

**这次事件的元教训**（值得刻在文件里）：

1. **AI 也会过载** —— 天蓝的运营 AI（飞书那边）跟人类一样，太多 trigger 会互相打断，产生"欠账感"（feedback-fragmented-work.md 里描述的现象在 AI 身上也成立）
2. **减负方式：先砍数量，再砍时段，最后砍原则** —— 顺序不能倒。原则（铁律）是最贵的，翻车之后重建成本远高于日常自查成本
3. **协作 AI 主动汇报"我卡住了"是好事** —— 说明它的 meta 层在工作。给它开这种"负反馈"通道，比等它默默翻车强
