# T005 · 归档规则讨论输入

**给天蓝 + 豆包 + GitHub Claude 一起过 · 决定归档规则应该怎么定**

**背景**: 手册说"每月人工判断归档",没给具体规则 → 每月要重新想一遍"这个该不该废" → 认知负担高

---

## 我(GitHub Claude)v1 提议 · 3 条规则

**Rule 1 · 明确废弃**
项目取消 / 方案推翻 / 天蓝说"这个思路不对" → 归档

**Rule 2 · 项目完成且 SOP 已沉淀**
某具体项目做完 · SOP 已经沉淀出通用版 · 原文件不再需要 → 归档

**Rule 3 · 90 天没动 + 无引用**
mtime > 90 天 且 MEMORY.md 里没提它 且 其他文件也没链它 → 归档

**归档不删** · 移到 archive/ 目录 · 顶部加 `【归档·已失效】` 标注 · 保留历史

---

## 我看到的潜在问题

### 问题 1 · Rule 3 的"无引用"判定不好做

- 我要能扫全库 `[[filename]]` 和 `](filename.md)` 的所有引用
- 飞书 base 里可能有引用我看不到
- 天蓝脑子里也可能有引用没写下来
- **误判风险**: 某文件其实是"稳定不变"(比如挂点环境限制)不是"失效",被 90 天规则冤枉

### 问题 2 · Rule 2 的"SOP 已沉淀"判定主观

- 谁说了算 = 通用 SOP 已经沉淀?
- 现有 SOP 覆盖 80% 情况可以吗? 还是必须 100%?

### 问题 3 · 没有"复活"机制

- 归档的文件万一以后又需要,怎么找回?
- 现在方案是 git log 翻,但天蓝不查 git

### 问题 4 · 3 条太少 or 太多?

- 手册原意是"规则最好越少越好"
- 但只有 Rule 1 明确废弃,其他 2 条都可能误判
- 是不是**只用 Rule 1 + 每月人工过一遍** 更保险?

## 备选方案(给豆包参考)

**方案 A · 严格 3 条**(我 v1)
- 优点: 机械可扫,不用每月想
- 缺点: Rule 3 误判风险

**方案 B · 只 Rule 1**
- 优点: 零误判
- 缺点: 只归档"明确说废"的,大量"其实过时了但没人说"的文件永远堆着

**方案 C · Rule 1 + 半年人工过一遍**
- 优点: 保底 + 定期扫
- 缺点: 半年一次容易忘

**方案 D · Rule 1 + 每月飞书出候选清单 + 天蓝 10min yes/no**
- 优点: 半自动化 · 决策成本低
- 缺点: 依赖飞书稳定 · 依赖天蓝每月记得看

**方案 E · 完全人工**
- 优点: 零风险
- 缺点: 认知负担全在天蓝

## 讨论要点(给豆包)

1. 归档规则应该"机械化"(有触发即归)还是"周期化"(每月扫一次)?
2. Rule 3(90 天 + 无引用)风险大不大?阈值可以调吗?
3. 有没有"复活"机制的必要?
4. 备选方案 A-E 哪个更契合天蓝的实际工作节奏?
